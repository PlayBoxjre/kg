1.1、注解开发规范：

1.1 .1 、行为型注解

  • DomainActivity：用于描述领域活动；该注解作用于类上。
属性：
id：领域活动id，符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二。
domainActivityName：领域活动名称，符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二，可以和id保持一致。
domainActivityDesc：领域活动描述,使用精简中文描述，建议不超过30字符。
示例：@DomainActivity(id = "saveOrderActivity", domainActivityName = "saveOrderActivity", domainActivityDesc = "订单保存")
  • DomainService：描述领域服务，该注解作用于类上。
 属性：
id:领域服务id、符合java变量命名规则，建议采用类名
首字母小写方式。要求全局独一无二。
domainServiceName:领域服务名称,符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二，可以和id保持一致。
domainServiceDes:领域服务描述、使用精简中文描述、建议不超过30字符。
例子：@DomainService(id="tradeOrderDomainService",domainServiceName = "tradeOrderDomainService",domainServiceDes = "订单领域服务")
  • DomainAbility：原子域能力，通过标记到具体的能力，在注册过程中进行扫描这个注解。该注解作用于类上。
属性：
id:领域服务id、符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二。
abilityName：能力名称,符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二，可以和id保持一致。
abilityDesc：能力描述，使用精简中文描述，建议不超过30字符。
示例：：@DomainAbility(id="createOrderAmountCheckAbility",abilityName ="createOrderAmountCheckAbility" ,abilityDesc = "订单金额校验")
  • DomainMethodDes：领域服务和能力的方法描述注解。该注解作用于方法上。
属性：
value：方法具体描述，使用精简中文描述，建议不超过30字符。
示例：@DomainMethodDesc("订单校验")

1.1.2 、资源型注解

  • EntityObject：实体描述注解，该注解作用于类上。
属性：
id：实体id、符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二。
entityName：实体名称,符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二，可以和id保持一致。
entityDesc：实体描述，使用精简中文描述，建议不超过30字符。
isAggRoot：是否聚合根、默认false。如果实体是聚合根则属性设置isAggRoot=true
示例：@EntityObject(id="tradeOrder",entityName = "tradeOrder",entityDesc = "订单实体",isAggRoot = true)
  • ValueObject：值对象描述注解，该注解作用于类上。
属性：
id：值对象id、符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二。
valueName：值对象名称,符合java变量命名规则，建议采用类名首字母小写方式。要求全局独一无二，可以和id保持一致。
valueDesc：实体描述，使用精简中文描述，建议不超过30字符。
示例：@ValueObject(id="address",valueName = "address",valueDesc = "地址值对象")

  • 实体类需标注@EntityObject注解

  • 包名格式为“com.公司名.项目名.模块名.……”。 如： com.midea.mkcloud.trade....

1.3 、分层架构规范

DDD 分层架构模型
  • 用户接口层/表现层(facade)：面向端或者前台提供服务适配，面向资源层提供资源适配。这一层聚集了接口适配相关的功能。
  • 应用层(application)：实现服务组合和编排，适应业务流程快速变化的需求。这一层聚集了应用服务和事件相关的功能。
  • 领域层(domain)：实现领域的核心业务逻辑。这一层聚集了领域模型的聚合、聚合根、实体、值对象、领域服务和事件等领域对象，以及它们组合所形成的业务能力。
  • 基础层(Infrastructure)：贯穿所有层，为各层提供基础资源服务。这一层聚集了各种底层资源相关的服务和能力。

1.3.1、二方api包

提供rpc调用所需api、数据信息或者消息事件等。如：springcloud如feign、dubbo的服务接口。
代码结构如图：
  • api: api接口类、必须以Api结尾
  • event: 事件相关信息
  • request: 请求信息封装、要求以Req结尾
  • response: 返回信息封装实体、要求以Resp结尾。

1.3.2、**facade（用户接口层）**

它主要存放用户接口层与前端交互、展现数据相关的代码。前端应用通过这一层的接口，向应用服务获取展现所需的数据。
这一层主要用来处理用户发送的 Restful 请求，解析用户输入的配置文件，并将数据传递给 Application 层。数据的组装、数据传输格式以及 Facade 接口等代码都会放在这一层目录里。
        facade的代码目录结构有：controller、service 两类。
代码结构如下：
  •  controller： springmvc controller类，必须以Controller结尾
  •  service: rpc调用实现，如果采用dubbo、hession之类框架，必须实现二方包中Api接口。 

 1.3.3、**application(应用层 )**

Application 的代码目录结构有：event 和 service。
代码结构如下：
  • convert:DTO对象和领域对象转换器
  • process: 流程相关根目录。下面包含activity子包、context子包。
    • activity:  活动包，存放所有流程活动的目录。下面的类必须标注@DomainActivity注解且遵从DomainActivity规范。类必须以Activity结尾。同时必须继承ActivityNode、ActivityCondNode其中之一。
    • context: 定义活动泛型参数。该包下类必须继承自BaseContext类。
  • service: 应用服务，应用服务会对多个领域服务或外部应用服务进行封装、编排和组合，对外提供粗粒度的服务。应用服务主要实现服务组合和编排，是一段独立的业务逻辑。你可以将所有应用服务放在一个应用服务类里，也可以把一个应用服务设计为一个应用服务类，以防应用服务类代码量过大。

1.3.4、**domain领域层** 

Domain 是由一个或多个聚合包构成，共同实现领域模型的核心业务逻辑。聚合内的代码模型是标准和统一的，包括：ability、dto、extension、model、infra、service等子目录
代码结构如下：
  •   ability:   能力类。必须标准@DomainAbility注解，遵守@DomainAbility规范。继承BaseAbility父类。类名以Ability结尾。每个能力方法需要标注@DomainMethodDesc注解。
  •   dto: rpc调用数据传输对象,必须以DTO结尾。
  •   infra: 基础设施相关
    • rpc:  rpc接口类存放，必须是接口且Service结尾。
    • repo：仓储接口，必须是接口且以Repo结尾。
  • extension: 扩展点接口，类必须以ExtSPI结尾。通用扩展点继承自GenericExtensionSPI类或者GroupExtensionSPI。禁止直接继承BaseExtensionSPI。
  • model: 
  • entity:  实体 、实体类必须标注@EntityObject且遵循EntityObject规范。继承BaseModel父类。实体必须以Entity结尾。所有*Entity类必须放置在'..entity'包下
  • vo:   值对象、值对象必须标注@ValueObject且遵循ValueObject规范。 值对象必须以VO结尾。所有 *VO类必须放置在'..vo'包下
  • service: 领域服务，所有领域服务必须标注@DomainService注解且遵从@DomainService规范，且必须以Service结尾。

1.3.5、Infrastructure**基础设施层**

Infrastructure 的代码目录结构有：dataobject、repo、rpc。
代码结构如下：
  • dataobject: 数据持久化对象
  • repo
    • convert: 领域对象和数据持久化对象转换器。转化器要求Converter结尾
    • mapper:  持久层dao接口，如mybtais 的mapper接口。该包下类必须接口，而且必须以mapper结尾
    • impl:   仓储实现类。该包下类必须为领域层.infra.repo实现类，而且类必须以RepoImpl结尾。并且标准 @Component注解
  • rpc:  rpc实现类。该包下类必须为领域层.infra.rpc实现类，而且类必须以ServiceImpl结尾。并且标准@Component注解

2、archeunit代码规约

2.1、分层检查


前端应用调用发布在 API 网关上的 Facade 服务，Facade 定向到应用服务。应用服务作为服务组织和编排者，
它的服务调用有这样两种路径：
第一种是应用服务调用并组装领域服务。此时领域服务会组装实体和实体方法，实现核心领域逻辑。领域服务通过仓储服务获取持久化数据对象，完成实体数据初始化。
第二种是应用服务直接调用仓储服务。这种方式主要针对像缓存、文件等类型的基础层数据访问。这类数据主要是查询操作，没有太多的领域逻辑，不经过领域层，不涉及数据库持久化对象。
ddd定义了四层 layer：
  1. 用户接口层facade) : facade主要是根据请求来源或类型进行适配，对app层的结果进行封装。不能被其他 Layer 调用.
  2. 应用层（application）：应用服务，对多个领域服务和外部微服务调用的封装，编排和组合，对用户接口层提供流程上的核心业务逻辑。 能够只能被 Facade 调用
  3. 领域层（domain）：主要是聚合的划分，包括聚合根、实体、值对象，以及领域能力和网关（rpc）接口 ，本地事务的管理 。仅能够被 Application 访问。
  4. 基础设施层（ Infrastructure ）：采用了依赖倒置的原则，infrastructure层依赖于domain层，是领域层持久以及网关(rpc)的具体实现。 只能application、domain层访问。
另外：
1、api层是对外提供的SDK，包含API的定义，以及数据传输对象。被facade和第三方应用依赖。
2、mbf框架基础包。mbf基础框架包统一以com.midea.mbf包开头的类。可以被任何层依赖。
3、扩展点插件包。扩展点统一com.midea.{场景}.{模块}.biz开头的类。只能被domain层依赖。
事件：
1、DDD中的事件分为两种类型：领域事件（Domain Events）和集成事件（Integration Events）。
2、领域事件：在同一个微服务内的聚合之间的事件传递。使用进程内的通信机制完成。可以使用Spring 事件发布机制
3、集成事件：跨微服务的事件传递。使用事件总线（EventBus）实现。借助rocketmq实现事件发布/订阅
DDD之领域服务、应用服务原则：
1、聚合中的实体中没有业务逻辑代码，只有对象的创建、对象的初始化、状态管理等个体相关的代码。
2、对于聚合内的业务逻辑，我们编写领域服务（Domain Service），而对于跨聚合协作以及聚合与外部系统协作的逻辑，我们编写应用服务（Application Service）。聚合之间协作通过领域事件来完成
3、应用服务协调多个领域服务、外部系统来完成一个用例。

2.2、数据视图依赖检查

  • 数据持久化对象 PO(Persistent Object)，与数据库结构一一映射，是数据持久化过程中的数据载体。
  • 领域对象 DO（Domain Object），微服务运行时的实体，是核心业务的载体。
  • 数据传输对象 DTO（Data Transfer Object），用于前端与应用层或者微服务之间的数据组装和传输，是应用之间数据传输的载体。
  • 视图对象 VO（View Object），用于封装展示层指定页面或组件的数据
基础层
基础层的主要对象是 PO 对象。我们需要先建立 DO 和 PO 的映射关系。当 DO 数据需要持久化时，仓储服务会将 DO 转换为 PO 对象，完成数据库持久化操作。当 DO 数据需要初始化时，仓储服务从数据库获取数据形成 PO 对象，并将 PO 转换为 DO，完成数据初始化。大多数情况下 PO 和 DO 是一一对应的。但也有 DO 和 PO 多对多的情况，在 DO 和 PO 数据转换时，需要进行数据重组。
领域层
领域层的主要对象是 DO 对象。DO 是实体和值对象的数据和业务行为载体，承载着基础的核心业务逻辑。通过 DO 和 PO 转换，我们可以完成数据持久化和初始化。
应用层
应用层的主要对象是 DO 对象。如果需要调用其它微服务的应用服务，DO 会转换为 DTO，完成跨微服务的数据组装和传输。用户接口层先完成 DTO 到 DO 的转换，然后应用服务接收 DO 进行业务处理。如果 DTO 与 DO 是一对多的关系，这时就需要进行 DO 数据重组。
用户接口层
用户接口层会完成 DO 和 DTO 的互转，完成微服务与前端应用数据交互及转换。Facade 服务会对多个 DO 对象进行组装，转换为 DTO 对象，向前端应用完成数据转换和传输。前端应用前端应用主要是 VO 对象。展现层使用 VO 进行界面展示，通过用户接口层与应用层采用 DTO 对象进行数据交互。

3、最佳实践

3.1、按照领域进行划分业务边界，在本领域内只能操作本领域对象，严谨直接操作其他领域的数据或者对象

对外部领域的访问可以通过以下两种方式进行，
  • 通过本领域发布标准事件方式通知其他领域（异步方式）
  • 通过调用外部领域的API来进行交互（同步方式）可以将其他领域看做是另外一个微服务，如果当前领域需要使用，可以在repo层做接口，实现对外部领域的调用。
严禁直接在当前领域操作另外一个领域的底层方法或者访问底层数据，比如在WHO领域里面，直接访问IO或者DO的数据库操作方法。可能导致的问题有：
  • 直接访问外部领域，外部领域不知道你做了什么，可能和外部领域的逻辑有冲突
  • 直接操作外部领域的数据库，没有分布式锁等，会导致数据覆盖错乱问题
  • 业务逻辑没有按照领域进行封装，高度耦合，增加了系统复杂度
这个原则非常重要，必须严格遵守，严格按照领域边界划分职责，业务逻辑高度封装、内聚在一个领域模型（聚合）内部。

3.2、分布式锁加到四层架构的应用层（APP层）

APP层是对领域服务的组合和编排，在APP层的每一个服务要明确的对当前领域对象的一个操作，比如评审履约单，出库确认，审核RFO等，这个时候要明确的在APP层对当前领域的对象进行加锁，以防止并发操作等问题。

3.3、领域能力不能嵌套，按照需求结构化的方式编排

能力都是在领域服务中进行编排，并打平，不允许多层嵌套，一个领域能力做一个原子的业务逻辑，不要嵌套。 也不要一个方法定义一个能力类，可以一个能力类写多个方法，多个方法按照业务功能命名（不要命名exceute等无意义的方法名）。

4.4、对标准领域事件(MQ消息)的监听逻辑，要在对应的领域内进行处理

一个领域(比如FOX)发出的标准事件后，会有多个领域进行监听（RFO/IO/TO），每个领域在监听这个事件后，业务逻辑要在各自的领域（RFO/IO/TO）进行处理，不要都放到发送事件的领域（FOX）进行处理，领域事件发送完成后，当前领域的工作已经完成，需要其他领域接力处理，要确保业务逻辑按照领域进行划分边界，是那个领域的业务，就在那个领域内进行处理。

4.5、消息的Event 要定义到API 层，不要写到domain层

本质上Event是当前领域对外通知的的一个DTO，是需要其他领域或者外部微服务应用来监听的，所以要通过二方包的DTO方式让其他领域或者外部应用能直接访问到。 如果直接定义到当前领域层，而不是API层，那其他领域还要自己反序列化消息体，自己去解析。所以把Event 定义到API层，如果有升级，加字段等方式，升级二方包的版本即可。

4.6、中台要定义标准的消息队列事件，一个应用服务完成后只允许发送一个事件

原则上一个服务，在领域对象状态更新后，只允许发布一个标准的Event， 即只发送一个Topic，其他领域，外部微服务应用或者场景前台进行监听，而不是为了每个场景或者某个业务都发送一个不同的Topic，这样会导致Topic 膨胀，也会使得中台代码复杂度增加，难以维护。

4.7、严格按照领域划分的边界，进行代码结构和包结构的编写

比如履约中心，有FOX，DOX，IOX等领域，要以这几个领域为边界，在Domain层进行领域划分，建立以领域为边界的package，不要随意增加package。将领域的所有核心业务都在这个package下完成。

4.8、代码层级要清晰，易懂，不要写见缝插针式的代码

要严格按照规范，确定代码结构正确的代码编写方式。否则因为中台的逻辑复杂，业务需求膨胀，最终会快速导致代码腐化，难以维护，降低开发效率。
禁止见缝插针随意写代码，比如在一个find方法中，正常是要查询一个对象，但是在find方法后写了一个兜底校验，虽然也能满足功能需求，但是会降低代码的可读性，也影响系统研发效率。
具体的包结构参考开发框架demo

4.9、正确使用和定义中台业务数据库表的标准字段和扩展字段

中台的定位是提供通用能力，所以在中台的领域模型以及数据库中，有涉及到新的需求，需要增加字段，首先要考虑添加字段的必要性。 如果是通用的字段，中台的通用能力，才考虑添加新的字段。 并且还要评估当前字段是否需要在数据库层面进行查询等。 如果是特定场景才需要的字段，或者只是一个标识,只是某个场景特有的信息，则不需要添加字段， 这个字段可以放到扩展字段extinfo 中， 放到extinfo 中也可以作为ES的检索条件来进行查询。
同时注意报错数据的长度问题，大段的业务信息，报错信息不允许全部记录到中台的数据库中。以下场景适合放到扩展字段，防止表的字段膨胀：
  • 一些边缘临时性、小场景、非通用的需求、仅仅程序代码处理时需要标志位字段、字段的值长度是可估计且比较短的，建议放到扩展字段
以下3种情况，不允许放到扩展字段：
  • 对于存储的内容的数量不可估计（譬如存放编码数组，且数组长度不定，存放了几百上千个开机编码值）
  • 字段的值长度很长（譬如把一些exception的堆栈信息、错误原因存储到扩展字段）
  • 对于核心、通用的业务意义的数据，必须显式定义业务字段，规范数据模型
大家要注意避免走向各自的极端：要不就是全部放扩展字段，要不就是使劲加表字段。
要知道扩展字段涉及的业务字段很多，不能因为长度爆了，导致影响大面积的各个功能报错！

4.10、禁止把一些非核心业务的附加需求逻辑放到核心业务逻辑中

业务代码中要清晰的实现业务逻辑，不要把一些非核心业务的附加需求逻辑放到核心业务逻辑中，比如同步ES的操作，对单据操作的业务审计日志等，都是非核心业务。 不要放到核心的业务逻辑中。

4.11、对业务对象进行操作（增删改）,只能信任和依赖数据库

对业务领域数据的操作，比如新增，修改，取消等操作，唯一能信任的就是数据库，ES，Redis，等NOSQL 数据库都不可以信任。尤其是ES搜索引擎，不能用来判断业务单据是否存在，单据的状态，获取单据信息用来判断等。  这是因为ES的单据可能存在更新延迟或者漏数据，如果用ES来判断就会对业务操作产生系统bug，逻辑错乱。

4.12、中台微服务设计要注意数据质量，在新增、保存单据时候严格做到数据质量合法性校验

业务中台新增的业务单据或者在进行数据编辑的时候，要对前端或者上游传入的数据合法性进行检查，比如判空校验，校验数据合法性（例如手机号校验），数据有效性（例如传入客户编码，需要验证客户编码在客户中心是否有效）等。

4.13、二方包在开发阶段使用_snapshot版本，在生产环境使用正式release版本,新版本要兼容历史版本

什么是二方包，即提供给其他微服务的api 接口打成的 jar 包，包括feign接口，对外暴露的常量，枚举，DTO等。二方包在开发阶段因为经常变化接口参数，常量等，可以使用snapshot版本，快速修改和适配新的参数和引入新的接口。在测试稳定后要改成正式版本（不带snapshot的版本）并通过devops打包稳定版本二方包推送到mvn仓库，并通知相关调用方改为正式版本，调用方进行最后验证，然后发布到生产环境。如果使用快照版本容易发生服务提供方最新修改未推送，或者最新修改不应该推送而进行了推送，导致调用方使用到不正确的版本，引起bug.
二方包每次的升级迭代要考虑接口的版本兼容性，每次升级原则上不强制调用方升级，可以允许多版本客户端并存。

2.1、分层检查

前端应用调用发布在 API 网关上的 Facade 服务，Facade 定向到应用服务。应用服务作为服务组织和编排者，
它的服务调用有这样两种路径：
第一种是应用服务调用并组装领域服务。此时领域服务会组装实体和实体方法，实现核心领域逻辑。领域服务通过仓储服务获取持久化数据对象，完成实体数据初始化。
第二种是应用服务直接调用仓储服务。这种方式主要针对像缓存、文件等类型的基础层数据访问。这类数据主要是查询操作，没有太多的领域逻辑，不经过领域层，不涉及数据库持久化对象。
ddd定义了四层 layer：
  1. 用户接口层facade) : facade主要是根据请求来源或类型进行适配，对app层的结果进行封装。不能被其他 Layer 调用.
  2. 应用层（application）：应用服务，对多个领域服务和外部微服务调用的封装，编排和组合，对用户接口层提供流程上的核心业务逻辑。 能够只能被 Facade 调用
  3. 领域层（domain）：主要是聚合的划分，包括聚合根、实体、值对象，以及领域能力和网关（rpc）接口 ，本地事务的管理 。仅能够被 Application 访问。
  4. 基础设施层（ Infrastructure ）：采用了依赖倒置的原则，infrastructure层依赖于domain层，是领域层持久以及网关(rpc)的具体实现。 只能application、domain层访问。
另外：
1、api层是对外提供的SDK，包含API的定义，以及数据传输对象。被facade和第三方应用依赖。
2、mbf框架基础包。mbf基础框架包统一以com.midea.mbf包开头的类。可以被任何层依赖。
3、扩展点插件包。扩展点统一com.midea.{场景}.{模块}.biz开头的类。只能被domain层依赖。
事件：
1、DDD中的事件分为两种类型：领域事件（Domain Events）和集成事件（Integration Events）。
2、领域事件：在同一个微服务内的聚合之间的事件传递。使用进程内的通信机制完成。可以使用Spring 事件发布机制
3、集成事件：跨微服务的事件传递。使用事件总线（EventBus）实现。借助rocketmq实现事件发布/订阅
DDD之领域服务、应用服务原则：
1、聚合中的实体中没有业务逻辑代码，只有对象的创建、对象的初始化、状态管理等个体相关的代码。
2、对于聚合内的业务逻辑，我们编写领域服务（Domain Service），而对于跨聚合协作以及聚合与外部系统协作的逻辑，我们编写应用服务（Application Service）。聚合之间协作通过领域事件来完成
3、应用服务协调多个领域服务、外部系统来完成一个用例。
2.2、数据视图依赖检查
数据持久化对象 PO(Persistent Object)，与数据库结构一一映射，是数据持久化过程中的数据载体。
领域对象 DO（Domain Object），微服务运行时的实体，是核心业务的载体。
数据传输对象 DTO（Data Transfer Object），用于前端与应用层或者微服务之间的数据组装和传输，是应用之间数据传输的载体。
视图对象 VO（View Object），用于封装展示层指定页面或组件的数据
基础层
基础层的主要对象是 PO 对象。我们需要先建立 DO 和 PO 的映射关系。当 DO 数据需要持久化时，仓储服务会将 DO 转换为 PO 对象，完成数据库持久化操作。当 DO 数据需要初始化时，仓储服务从数据库获取数据形成 PO 对象，并将 PO 转换为 DO，完成数据初始化。大多数情况下 PO 和 DO 是一一对应的。但也有 DO 和 PO 多对多的情况，在 DO 和 PO 数据转换时，需要进行数据重组。
领域层
领域层的主要对象是 DO 对象。DO 是实体和值对象的数据和业务行为载体，承载着基础的核心业务逻辑。通过 DO 和 PO 转换，我们可以完成数据持久化和初始化。
应用层
应用层的主要对象是 DO 对象。如果需要调用其它微服务的应用服务，DO 会转换为 DTO，完成跨微服务的数据组装和传输。用户接口层先完成 DTO 到 DO 的转换，然后应用服务接收 DO 进行业务处理。如果 DTO 与 DO 是一对多的关系，这时就需要进行 DO 数据重组。
用户接口层
用户接口层会完成 DO 和 DTO 的互转，完成微服务与前端应用数据交互及转换。Facade 服务会对多个 DO 对象进行组装，转换为 DTO 对象，向前端应用完成数据转换和传输。前端应用前端应用主要是 VO 对象。展现层使用 VO 进行界面展示，通过用户接口层与应用层采用 DTO 对象进行数据交互。

3、最佳实践
3.1、按照领域进行划分业务边界，在本领域内只能操作本领域对象，严谨直接操作其他领域的数据或者对象
对外部领域的访问可以通过以下两种方式进行，
通过本领域发布标准事件方式通知其他领域（异步方式）
通过调用外部领域的API来进行交互（同步方式）可以将其他领域看做是另外一个微服务，如果当前领域需要使用，可以在repo层做接口，实现对外部领域的调用。
严禁直接在当前领域操作另外一个领域的底层方法或者访问底层数据，比如在WHO领域里面，直接访问IO或者DO的数据库操作方法。可能导致的问题有：
直接访问外部领域，外部领域不知道你做了什么，可能和外部领域的逻辑有冲突
直接操作外部领域的数据库，没有分布式锁等，会导致数据覆盖错乱问题
业务逻辑没有按照领域进行封装，高度耦合，增加了系统复杂度
这个原则非常重要，必须严格遵守，严格按照领域边界划分职责，业务逻辑高度封装、内聚在一个领域模型（聚合）内部。
3.2、分布式锁加到四层架构的应用层（APP层）
APP层是对领域服务的组合和编排，在APP层的每一个服务要明确的对当前领域对象的一个操作，比如评审履约单，出库确认，审核RFO等，这个时候要明确的在APP层对当前领域的对象进行加锁，以防止并发操作等问题。
3.3、领域能力不能嵌套，按照需求结构化的方式编排
能力都是在领域服务中进行编排，并打平，不允许多层嵌套，一个领域能力做一个原子的业务逻辑，不要嵌套。 也不要一个方法定义一个能力类，可以一个能力类写多个方法，多个方法按照业务功能命名（不要命名exceute等无意义的方法名）。
4.4、对标准领域事件(MQ消息)的监听逻辑，要在对应的领域内进行处理
一个领域(比如FOX)发出的标准事件后，会有多个领域进行监听（RFO/IO/TO），每个领域在监听这个事件后，业务逻辑要在各自的领域（RFO/IO/TO）进行处理，不要都放到发送事件的领域（FOX）进行处理，领域事件发送完成后，当前领域的工作已经完成，需要其他领域接力处理，要确保业务逻辑按照领域进行划分边界，是那个领域的业务，就在那个领域内进行处理。
4.5、消息的Event 要定义到API 层，不要写到domain层
本质上Event是当前领域对外通知的的一个DTO，是需要其他领域或者外部微服务应用来监听的，所以要通过二方包的DTO方式让其他领域或者外部应用能直接访问到。 如果直接定义到当前领域层，而不是API层，那其他领域还要自己反序列化消息体，自己去解析。所以把Event 定义到API层，如果有升级，加字段等方式，升级二方包的版本即可。
4.6、中台要定义标准的消息队列事件，一个应用服务完成后只允许发送一个事件
原则上一个服务，在领域对象状态更新后，只允许发布一个标准的Event， 即只发送一个Topic，其他领域，外部微服务应用或者场景前台进行监听，而不是为了每个场景或者某个业务都发送一个不同的Topic，这样会导致Topic 膨胀，也会使得中台代码复杂度增加，难以维护。
4.7、严格按照领域划分的边界，进行代码结构和包结构的编写
比如履约中心，有FOX，DOX，IOX等领域，要以这几个领域为边界，在Domain层进行领域划分，建立以领域为边界的package，不要随意增加package。将领域的所有核心业务都在这个package下完成。
4.8、代码层级要清晰，易懂，不要写见缝插针式的代码
要严格按照规范，确定代码结构正确的代码编写方式。否则因为中台的逻辑复杂，业务需求膨胀，最终会快速导致代码腐化，难以维护，降低开发效率。
禁止见缝插针随意写代码，比如在一个find方法中，正常是要查询一个对象，但是在find方法后写了一个兜底校验，虽然也能满足功能需求，但是会降低代码的可读性，也影响系统研发效率。
具体的包结构参考开发框架demo
4.9、正确使用和定义中台业务数据库表的标准字段和扩展字段
中台的定位是提供通用能力，所以在中台的领域模型以及数据库中，有涉及到新的需求，需要增加字段，首先要考虑添加字段的必要性。 如果是通用的字段，中台的通用能力，才考虑添加新的字段。 并且还要评估当前字段是否需要在数据库层面进行查询等。 如果是特定场景才需要的字段，或者只是一个标识,只是某个场景特有的信息，则不需要添加字段， 这个字段可以放到扩展字段extinfo 中， 放到extinfo 中也可以作为ES的检索条件来进行查询。
同时注意报错数据的长度问题，大段的业务信息，报错信息不允许全部记录到中台的数据库中。以下场景适合放到扩展字段，防止表的字段膨胀：
一些边缘临时性、小场景、非通用的需求、仅仅程序代码处理时需要标志位字段、字段的值长度是可估计且比较短的，建议放到扩展字段
以下3种情况，不允许放到扩展字段：
对于存储的内容的数量不可估计（譬如存放编码数组，且数组长度不定，存放了几百上千个开机编码值）
字段的值长度很长（譬如把一些exception的堆栈信息、错误原因存储到扩展字段）
对于核心、通用的业务意义的数据，必须显式定义业务字段，规范数据模型
大家要注意避免走向各自的极端：要不就是全部放扩展字段，要不就是使劲加表字段。
要知道扩展字段涉及的业务字段很多，不能因为长度爆了，导致影响大面积的各个功能报错！
4.10、禁止把一些非核心业务的附加需求逻辑放到核心业务逻辑中
业务代码中要清晰的实现业务逻辑，不要把一些非核心业务的附加需求逻辑放到核心业务逻辑中，比如同步ES的操作，对单据操作的业务审计日志等，都是非核心业务。 不要放到核心的业务逻辑中。
4.11、对业务对象进行操作（增删改）,只能信任和依赖数据库
对业务领域数据的操作，比如新增，修改，取消等操作，唯一能信任的就是数据库，ES，Redis，等NOSQL 数据库都不可以信任。尤其是ES搜索引擎，不能用来判断业务单据是否存在，单据的状态，获取单据信息用来判断等。 这是因为ES的单据可能存在更新延迟或者漏数据，如果用ES来判断就会对业务操作产生系统bug，逻辑错乱。
4.12、中台微服务设计要注意数据质量，在新增、保存单据时候严格做到数据质量合法性校验
业务中台新增的业务单据或者在进行数据编辑的时候，要对前端或者上游传入的数据合法性进行检查，比如判空校验，校验数据合法性（例如手机号校验），数据有效性（例如传入客户编码，需要验证客户编码在客户中心是否有效）等。
4.13、二方包在开发阶段使用_snapshot版本，在生产环境使用正式release版本,新版本要兼容历史版本
什么是二方包，即提供给其他微服务的api 接口打成的 jar 包，包括feign接口，对外暴露的常量，枚举，DTO等。二方包在开发阶段因为经常变化接口参数，常量等，可以使用snapshot版本，快速修改和适配新的参数和引入新的接口。在测试稳定后要改成正式版本（不带snapshot的版本）并通过devops打包稳定版本二方包推送到mvn仓库，并通知相关调用方改为正式版本，调用方进行最后验证，然后发布到生产环境。如果使用快照版本容易发生服务提供方最新修改未推送，或者最新修改不应该推送而进行了推送，导致调用方使用到不正确的版本，引起bug.
二方包每次的升级迭代要考虑接口的版本兼容性，每次升级原则上不强制调用方升级，可以允许多版本客户端并存。>)
