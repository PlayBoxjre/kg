一、 概述

在了解和学习领域驱动设计思想和一般分析步骤之后，需要落实到具体项目中，完成开发环境的全流程实施。其中包括技术选型、 项目包/模块设计、项目组织结构、开发规范等落地实施中的流程工作。
本文将根据领域驱动设计思想的学习，提供一种根据领域驱动的微服务架构项目落地解决方案。
希望能够提供给自己一个脚手架项目，学习和应用领域驱动设计和微服务架构。提供一些实际项目启动环境准备的实施经验，更深理解熟悉理论到实践中的碰到的各种问题和解决方法。

1. 1 为什么使用DDD

Quote
领域驱动设计可以很有效的指导微服务的拆分，设计和实现. 通过合理的拆分微服务，分层结构可以有效的实现系统复杂性的治理，构建高内聚，低耦合，可演进的分布式系统.

1.1.1 微服务的拆分

当前系统在从传统单体的架构想微服务转型过程中，一直都没有一个合理，有效的拆分微服务的理论，都是按照业务功能模块的方式去拆分微服务， 当前的拆分方式只是将一个大的单体拆分为一个小的单体，虽然能实现模块间发布解耦，水平扩展等大单体面临的问题，但是还是存在微服务业务架构拆分不清晰，代码实现混乱的问题. 主要有下面几个问题：
  • 微服务的拆分没有理论基础：
    按照功能模块的方式进行拆分，没有理论指导，粒度拆分过大或者过细.
  • 微服务的拆分粒度太细：
    项目复杂度变大，太多分布式系统运维和维护变得更复杂.
  • 微服务的拆分粒度过大：
    系统不解耦，耦合严重，成为大的单体.
Warning
结合领域驱动设计方法：可以合理拆分微服务，指导微服务的拆分，按照领域边界设计微服务，做到领域模型就是微服务的设计和实现，真正实现高内聚低耦合的微服务系统

1.1.2 微服务的设计、实现和演进

传统的代码分层方式
  • 以数据模型为基础构建应用
    缺少和业务领域的映射，开发技术和业务产品需求语言无法统一.
  • 业务逻辑层过于复杂Service manager 承载所有业务逻辑，业务逻辑零散，结构不清晰.
  • 业务领域过于关注技术实现：
    业务开发要关注技术底层的设计与实现
Warning
基于DDD的微服务实现：在战术设计中会有聚合、聚合根、实体、值对象、领域服务、领域事件、应用服务和仓储等领域对象，这些领域对象在设计完成后，会以代码的形式映射到微服务中，按照领域边界拆分微服务，按照领域模型设计与实现微服务代码，按照分层架构，真正做到代码与设计保持高度一致,每一层代码逻辑清晰，保持可演进的架构.

二、项目应用架构和规范

2.1 分层架构

分层架构

分层架构

软件系统有各种各样的划分方式，但是根据软件行业的经验和惯例，普遍采用 LAYERED ARCHITECTURE
LAYERED ARCHITECTURE的基本原则是层中的任何元素都仅依赖于本层的其他元素或其下层的元素。向上的通信必须通过间接的方式进行

分层是属于大粒度的职责划分，太粗，我们有必要往下再down一层，细化到包结构的粒度，才能更好的指导我们的工作。

2.2 应用架构包结构设计

2.2.1 一般项目架构

上图，是我们MVC单体架构模式下的一般开发过程和分包分模块。一个工程贯穿了代码开发全部过程。即使我们将一个大的单体应用拆分，其应用设计架构和实际代码组织架构依然混淆业务和功能。使得业务被分散到各个服务中。
业务被分散到CURD操作中。跟踪和维护成本以后后续人员参与成本非常大。并且业务点很难识别和理解。细节分散到多个类或接口中。内聚性也比较差。

2.2.2 项目模块设计

微服务架构下，如果一个限界上下文·绑定一个微服务，那么微服务的数量以及部署成本都会很高。一般的实践，是一个微服务边界包含多个相关的限界上下文，组成一个微服务程序。微服务间使用rpc技术进行沟通。
内部限界上下文之间调用依然需要保证内聚性，不能直接交叉直接使用，限界上下文提供统一入口供其他上下文接入使用。可以以后拆分扩展成新的微服务。
模块设计按照DDD中的四层结构分层理念进行映射设计即可。
模块说明
模块 作用 描述
interface 用户交互 该层包含与其他系统/客户进行交互的接口与通信设施，在多数应用里，该层可能提供包括web services、rmi或rest等在内的一种或多种通信接口。该层主要由facade、dto和converter三类组件构成，三类组件均是典型的j2ee模式
app 业务协调组装 应用层通过应用服务接口来暴露系统的全部功能。在应用服务的实现中，它负责编排和转发，它将要实现的功能委托给一个或多个领域对象来实现，它本身只负责处理业务用例的执行顺序以及结果的拼装。通过这样一种方式，它隐藏了领域层的复杂性及其内部实现机制**。应用层没有任何的业务逻辑代码，它很简单，它主要为程序提供任务处理, 只负责协调并委派业务逻辑给领域对象进行处理。
domain 业务 域层是整个系统的核心层，该层维护一个使用面向对象技术实现的领域模型，是具体的业务领域层，是发生业务变化最为频繁的地方，是业务系统最核心的一层，是DDD关注的焦点和难点。domain层包含entity（实体）、valueobject(值对象)、domain event（领域事件）、domain sevice（领域服务）和repository（仓储）等多种重要的领域组件。
infras 业务依赖基础能力 基础设施层为interfaces、application和domain三层提供支撑。所有与具体平台、框架相关的实现会在infrastructure中提供，避免三层特别是domain层掺杂进这些实现，从而“污染”领域模型。infrastructure中最常见的一类设施是对象持久化的具体实现。例如：数据库操作ORM框架，缓存操作，消息队列操作，OSS调用，RPC调用等等
common 通用工具类（如dto） 数据传输对象包括DTO,VO,Parmas等从网络协议到infras之间的所有参数或结果对象
api 微服务和外围系统api定义 服务外抛接口定义的二方包，提供给服务调用方引入的依赖包。主要包含：服务接口定义，服务熔断降级处理类，出入参结构定义，事件定义等
message 消息处理消费应用 非api接入场景，通过消息消费，完成业务处理
scheduler 定时任务应用 非api接入场景，通过定时任务触发业务流程处理

2.2.3 项目包设计

按照领域和功能进行分包设计。保证在领域驱动模型设计思想的指导下，指导软件代码开发映射的一致性。
这是细粒度的拆分，也就是代码被组织的最小粒度。理解每一个包的含义，将开发过程设计的类放到对应包下归类。
